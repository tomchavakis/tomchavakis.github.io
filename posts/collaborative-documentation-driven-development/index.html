Collaborative Documentation Driven Development

Posted on Dec 18, 2020

Introduction

Collaborative Documentation Driven Development (CDDD), is an agile way of working within a software development team.

Every agile team in the software industry has to create software products, versions, and releases are released quickly and frequently.

This invention occurred because the software teams start the development phase without analyzing in depth what is going to be delivered.

Here comes this process, in which you have to document the technical implementation. This document will be under review from other team members until all the technical details will be clear and acceptable to everyone.

When

Before the end of each sprint, and after you have successfully delivered the tasks of the current sprint you have to make a preparation for the next sprint. Grooming meetings are part of this process.

The output of that meeting(s) is that the team will have the chance to adjust the business need and provide a high-level estimation. As a result, the Product Owner of the team will refine the backlog according to the team dynamics the velocity of this team.

At this particular moment, and after the grooming meeting, the team can take the time to analyze each task of the sprint and write a technical specification that will become a product of continuous collaboration with other team members.

The assignee will present the technical specification to other team members during the HOW meeting. Later on, the team will be based on this document for the development phase.

How

Step 1: Analyze the task at the Grooming Meeting

Step 2: Assign the Task to a team member

Step 3: Create a new Folder at the docs/dspec/ticket_number

Step 4: The assignee will generate a new MarkDown Document (readme.md) like this one:

docs/dspec/ticket_number/pb-13222/readme.md

Step 5: The assignee will complete the technical analysis document according to the template.

Step 6: Open a new PR

Step 7: The proposal is under scrutinization from all the participants

Step 8: When this PR will take approvals then is under development phase.

Advantages

  • All team members participate in technical decisions.
  • You create the technical documentation incrementally and not once-off.
  • This technique is supplementary of the Architecture Decision Records ADR
  • You link the Business Need with the Technical Implementation with the Ticket. It would be useful for other teams with the same business need.
  • You think before start coding. There are less possibilities to follow the wrong path.
  • A possible project handover will be easier because everything are documented.
  • More efficient communication and collaboration with other crafts. This document is the source of truth and other crafts can based on your solution.
  • Sprint Cohesion and continuous development

Disadvantages

  • It’s a more time-consuming method

Template

The template can be different between teams, because each team has very different structure and business operation. An example will be the following:

## Ticket Link

## Problem 
Describe the problem or business need.

## Proposed Solution
Describe the solution and the technical approach to the problem.

### Artifacts
Import the Diagram that justifies the solution.

### Endpoint(s)
Analyze the possible endpoint request structure with the relative payload and method. 
Analyze all the success and error response(s).

### Database Changes
Write down all the possible changes to the database 

### Event Message Payload
Analyze the possible event message at the relative topic. 

## Affected Components/Services
Describe all the affected components or other microservices that will be impacted by this solution.

## Attention needed
Recognize possible code refactorings or important things that should be communicated before the implementation. This will help the code reviewers.

## Next Steps/ Improvements
Write down the possible improvements and the remaining work.
